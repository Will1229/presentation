Recruiter Screen

30 - 45 min Video / Phone call Non-technical

The first touchpoint. Recruiters assess culture fit, product awareness, and motivation. They increasingly look for genuine Spotify product knowledge, not just "I like music." Be ready to discuss specific features, your compensation expectations, and your career goals.

Tip: Actually use Spotify for a week before this call. Explore Discover Weekly, Blend, AI DJ, and podcast features. Form opinions on what works and what could be improved.

Q Explain the difference between TCP and UDP. When would you use each? Medium ▾

Suggested Answer

TCP (Transmission Control Protocol): Connection-oriented, guarantees delivery and ordering. Uses a 3-way handshake, retransmission, flow control, and congestion control. Higher latency due to acknowledgments.

UDP (User Datagram Protocol): Connectionless, no delivery guarantee, no ordering. Lower latency, less overhead. Fire-and-forget.

When to use TCP: HTTP APIs, database connections, file transfers -- anything where data integrity matters.

When to use UDP: Real-time audio/video streaming, DNS lookups, gaming -- where speed matters more than occasional packet loss.

Spotify-relevant: Spotify uses TCP for API calls and metadata fetching, but their audio streaming protocol likely uses a custom protocol over TCP with chunked transfer and buffering, since losing audio packets creates audible artifacts. However, real-time features like "listening along" could benefit from UDP-based protocols.

Analysis: This is a frequently reported question in Spotify's tech screen. Don't just recite definitions -- connect it to real-world systems. Mentioning Spotify's own streaming architecture shows product awareness. Bonus: discuss QUIC (HTTP/3) as a modern alternative that combines UDP's speed with TCP's reliability.
Q How does a HashMap work internally? How do you handle collisions? Medium ▾

Suggested Answer

A HashMap stores key-value pairs in an array of buckets. The key's hashCode() is computed and mapped to a bucket index via hash % capacity.

Collision strategies:

  • Separate chaining: Each bucket holds a linked list (or tree in Java 8+ when chain length > 8) of entries. O(1) average, O(n) worst case.
  • Open addressing: On collision, probe the next slot (linear probing, quadratic probing, or double hashing). Better cache locality but degrades at high load factors.

Java specifics: Java's HashMap uses separate chaining. Default capacity is 16, load factor is 0.75. When load factor is exceeded, the map rehashes (doubles capacity). Since Java 8, long chains convert to red-black trees for O(log n) worst case.

Analysis: Spotify has asked this in multiple reported interviews. Go beyond the textbook: mention how bad hash functions cause hot buckets, how equals() and hashCode() contract violations cause bugs, and how ConcurrentHashMap handles thread safety (segment locking in Java 7, CAS + synchronized in Java 8+).
Q Coding: Two Sum / Pairs summing to target Easy ▾

Suggested Answer

Given an array and a target, find all pairs that sum to the target.

def two_sum_pairs(nums, target):
    seen = set()
    result = set()
    for num in nums:
        complement = target - num
        if complement in seen:
            result.add((min(num, complement), max(num, complement)))
        seen.add(num)
    return list(result)

# Example: two_sum_pairs([1, 99, 50, 50, 3, 97], 100)
# Returns: [(1, 99), (50, 50), (3, 97)]

Time: O(n) | Space: O(n)

For the classic single-pair variant returning indices, use a HashMap mapping value to index.

Analysis: This exact question ("sumTo100 -- listing all pairs that sum to 100") has been reported in Spotify's tech screen. It's easy but they watch for: handling duplicates, edge cases (empty array, single element), and code cleanliness. Write clean code first, then optimize. Talk through your approach before coding.
Q Coding: String Compression Easy ▾

Suggested Answer

Given a string of consecutive repeated characters, compress it: "aaabbc" becomes "a3b2c1".

def compress(s):
    if not s:
        return ""
    result = []
    count = 1
    for i in range(1, len(s)):
        if s[i] == s[i - 1]:
            count += 1
        else:
            result.append(s[i - 1] + str(count))
            count = 1
    result.append(s[-1] + str(count))
    compressed = "".join(result)
    return compressed if len(compressed) < len(s) else s

# compress("aaabbc") -> "a3b2c1"
# compress("abc") -> "abc" (no benefit)

Time: O(n) | Space: O(n)

Analysis: Reported in multiple Spotify interviews. Key edge cases: single characters, all same characters, empty string, and the optimization of returning the original if compressed isn't shorter. In-place variant (LeetCode 443) is also fair game -- modifying the input array directly for O(1) extra space.
Q What is a MITM attack and how do you prevent it? Medium ▾

Suggested Answer

A Man-in-the-Middle attack occurs when an attacker intercepts communication between two parties, potentially reading or altering messages.

Prevention:

  • TLS/HTTPS: Encrypt all traffic. Use certificate pinning for mobile apps to prevent proxy-based MITM.
  • Mutual TLS (mTLS): Both client and server authenticate each other. Common in service-to-service communication.
  • HSTS: Force HTTPS and prevent protocol downgrade attacks.
  • Certificate transparency: Detect rogue certificates.

Spotify context: With 600M+ users on public WiFi, MITM is a real threat to their mobile apps. Spotify likely uses certificate pinning in their apps and mTLS between internal microservices.

Analysis: Security questions are common for senior roles. Don't just list countermeasures -- explain the attack flow (ARP spoofing, DNS poisoning, rogue access points) and why each countermeasure addresses a specific attack vector. Mentioning certificate pinning specifically is a strong signal.

Onsite: Coding / Algorithms & Data Structures

45 - 60 min CoderPad live coding 1 - 2 problems

The main coding round. Expect 1-2 problems of medium-to-hard difficulty. Some candidates report this as the hardest round. Problems are often Spotify-themed (playlists, songs, streams). You must write running code and think out loud. Interviewers care as much about your process as your solution.

Tip: Focus your LeetCode practice on HashMaps, Heaps, Sorting, String manipulation, Two-pointer, and Sliding Window. Spotify has asked problems similar to LC 215 (Kth Largest), LC 295 (Find Median from Data Stream), and LC 347 (Top K Frequent).
Q Find the Kth largest element in an unsorted array Medium ▾

Suggested Answer

Three approaches, each with different tradeoffs:

1. Min-heap (optimal for streaming):

import heapq

def find_kth_largest(nums, k):
    heap = nums[:k]
    heapq.heapify(heap)        # O(k)
    for num in nums[k:]:       # O((n-k) log k)
        if num > heap[0]:
            heapq.heapreplace(heap, num)
    return heap[0]

Time: O(n log k) | Space: O(k)

2. Quickselect (optimal average case):

import random

def find_kth_largest(nums, k):
    target = len(nums) - k

    def quickselect(left, right):
        pivot_idx = random.randint(left, right)
        nums[pivot_idx], nums[right] = nums[right], nums[pivot_idx]
        pivot = nums[right]
        store = left
        for i in range(left, right):
            if nums[i] < pivot:
                nums[i], nums[store] = nums[store], nums[i]
                store += 1
        nums[store], nums[right] = nums[right], nums[store]
        if store == target:
            return nums[store]
        elif store < target:
            return quickselect(store + 1, right)
        else:
            return quickselect(left, store - 1)

    return quickselect(0, len(nums) - 1)

Time: O(n) average, O(n^2) worst | Space: O(1)

Analysis: This is LeetCode 215, reported in Spotify interviews. Start with the heap approach -- it's simpler and works for streaming data (relevant for Spotify's "most played songs" features). Mention Quickselect as the optimal average-case solution but note its worst-case. In an interview, the heap solution is usually sufficient and less error-prone to implement.
Q Find Median from Data Stream Hard ▾

Suggested Answer

Design a data structure that supports adding numbers and finding the median efficiently. Use two heaps: a max-heap for the lower half and a min-heap for the upper half.

import heapq

class MedianFinder:
    def __init__(self):
        self.lo = []  # max-heap (negate values)
        self.hi = []  # min-heap

    def add_num(self, num):
        heapq.heappush(self.lo, -num)
        heapq.heappush(self.hi, -heapq.heappop(self.lo))
        if len(self.hi) > len(self.lo):
            heapq.heappush(self.lo, -heapq.heappop(self.hi))

    def find_median(self):
        if len(self.lo) > len(self.hi):
            return -self.lo[0]
        return (-self.lo[0] + self.hi[0]) / 2.0

addNum: O(log n) | findMedian: O(1) | Space: O(n)

Analysis: LeetCode 295 -- reported as a Spotify Hard. The two-heap approach is elegant and the expected solution. Spotify relevance: calculating median listen time, median song duration, or real-time percentile metrics. Follow-up: "What if 99% of numbers are in [0, 100]?" Answer: use bucket sort / counting array for O(1) median.
Q Sort playlists and retrieve most played songs (Heap) Medium ▾

Suggested Answer

Given a list of songs with play counts, return the top K most played songs.

import heapq
from collections import Counter

def top_k_songs(play_history, k):
    counts = Counter(play_history)
    return heapq.nlargest(k, counts.keys(), key=counts.get)

# Example
plays = ["Shape of You", "Blinding Lights", "Shape of You",
         "Blinding Lights", "Blinding Lights", "Bad Guy"]
print(top_k_songs(plays, 2))
# Output: ["Blinding Lights", "Shape of You"]

For a more hands-on approach showing understanding of heaps:

def top_k_songs_manual(play_history, k):
    counts = Counter(play_history)
    min_heap = []
    for song, count in counts.items():
        heapq.heappush(min_heap, (count, song))
        if len(min_heap) > k:
            heapq.heappop(min_heap)
    return [song for count, song in sorted(min_heap, reverse=True)]

Time: O(n log k) | Space: O(n + k)

Analysis: This is essentially LeetCode 347 (Top K Frequent Elements) but Spotify-themed. The first approach uses Python's built-in nlargest -- practical and clean. The second shows heap mechanics. In the interview, present both: one for production code, one to prove you understand the data structure. Mention that Quickselect can achieve O(n) average for this too.
Q Check if two strings are anagrams Easy ▾

Suggested Answer

from collections import Counter

def is_anagram(s, t):
    return Counter(s) == Counter(t)

# O(n) time, O(1) space (fixed charset)

For a lower-level approach without Counter:

def is_anagram(s, t):
    if len(s) != len(t):
        return False
    counts = [0] * 26
    for c in s:
        counts[ord(c) - ord('a')] += 1
    for c in t:
        counts[ord(c) - ord('a')] -= 1
    return all(c == 0 for c in counts)
Analysis: This is a warm-up question -- don't overthink it. But handle edge cases cleanly: case sensitivity, Unicode characters, empty strings. In a Spotify context, this could relate to fuzzy search in song titles. Show you can write clean, bug-free code quickly and move on.
Q Design a Playlist class with add, remove, shuffle, and sort operations Medium ▾

Suggested Answer

import random

class Playlist:
    def __init__(self, name):
        self.name = name
        self.songs = []          # ordered list
        self.song_set = set()    # O(1) lookup

    def add(self, song_id):
        if song_id not in self.song_set:
            self.songs.append(song_id)
            self.song_set.add(song_id)

    def remove(self, song_id):
        if song_id in self.song_set:
            self.songs.remove(song_id)
            self.song_set.discard(song_id)

    def shuffle(self):
        # Fisher-Yates shuffle -- O(n), uniform distribution
        for i in range(len(self.songs) - 1, 0, -1):
            j = random.randint(0, i)
            self.songs[i], self.songs[j] = self.songs[j], self.songs[i]

    def get_top_k(self, play_counts, k):
        return sorted(self.songs,
                       key=lambda s: play_counts.get(s, 0),
                       reverse=True)[:k]
Analysis: OOP design questions test your ability to write clean, extensible code. Key points: use a set for O(1) membership checks alongside a list for ordering. Implement Fisher-Yates shuffle (not naive random swap) to ensure uniform distribution -- Spotify's actual shuffle algorithm has been a famous engineering challenge. Discuss: "Spotify's users perceived their random shuffle as 'not random enough' because truly random sequences have clusters. Spotify engineered a 'spread' algorithm instead."

Onsite: System Design

60 min Whiteboard / Virtual Dialogue-heavy

The most challenging round according to recent candidates. You'll design a large-scale backend system, usually Spotify-themed. This round is heavily dialogue-driven -- ask clarifying questions, discuss tradeoffs, and drive the conversation. Spotify values broad understanding of the system design process and domain expertise.

Tip: Study Spotify's actual architecture: microservices on GCP/Kubernetes, event-driven with Kafka/PubSub, edge caching via CDN, gRPC for internal service communication. Key numbers: 600M+ users, 100M+ songs, 5+ billion playlists, ~1 billion streams per day.
Q Design a Music Streaming Service Hard ▾

Suggested Answer

Requirements

Functional: Search songs/artists/albums, stream music in real-time, create/manage playlists, personalized recommendations. Non-functional: Sub-200ms playback start, 99.99% availability, support 100M+ concurrent users.

Back-of-Envelope

  • 600M users, 200M DAU, avg 10 songs/day = 2B streams/day
  • Average song = 5MB, 100M songs = 500TB storage
  • Peak QPS: 2B / 86400 * 3 (peak multiplier) ~ 70K streams/sec
  • Bandwidth: 70K * 320kbps = ~22 Gbps peak

High-Level Architecture

  • API Gateway: Rate limiting, auth, routing. REST for client APIs, gRPC for internal services.
  • Song Service: Metadata CRUD. PostgreSQL with read replicas. Redis cache for hot songs.
  • Streaming Service: Chunked delivery. Songs split into 5-10 second chunks. Client prefetches next chunk during playback.
  • CDN Layer: Multi-tier: edge PoPs for popular songs, regional for long-tail. ~80% cache hit rate.
  • Search Service: Elasticsearch cluster. Inverted index on song name, artist, album, lyrics. Fuzzy matching for typo tolerance.
  • Recommendation Service: Collaborative filtering + content-based. Batch processing (Spark) for model training, real-time for serving.
  • Playlist Service: NoSQL (Cassandra) for high write throughput. Playlist = ordered list of song IDs.

Key Design Decisions

  • Audio format: Store in multiple bitrates (96, 160, 320 kbps). Adaptive bitrate streaming based on network quality.
  • Object storage: S3/GCS for audio files. CDN pulls from origin on cache miss.
  • Playback start optimization: Prefetch first chunk + buffer. Use DNS prefetching and connection pooling.
Analysis: This is the most common system design question. Drive the conversation by asking: "Should I focus on streaming architecture or recommendation system?" Start with requirements, do the math, draw the architecture, then deep-dive into 1-2 areas. The CDN + chunked streaming architecture is the most interesting part -- spend time there. Don't try to cover everything; depth in one area beats breadth across all.
Q Design a Music Recommendation Engine Hard ▾

Suggested Answer

Signal Collection

  • Explicit signals: Likes, saves, playlist adds, follows
  • Implicit signals: Play count, skip rate, listen duration, time of day, repeat plays
  • Negative signals: Skip within 5s, "hide song", unlike

Recommendation Approaches

  • Collaborative filtering: "Users similar to you also liked X." Matrix factorization on the user-song interaction matrix. Cold start problem for new users/songs.
  • Content-based filtering: Audio features (tempo, key, energy, danceability), genre tags, artist similarity graph. Handles cold start for new songs.
  • Hybrid approach: Combine both with a weighted ensemble. Use collaborative filtering for established users, content-based for new users/songs.

Architecture

  • Batch pipeline: Daily model training on full dataset (Spark/Dataflow). Generates user embeddings and song embeddings stored in a vector DB.
  • Real-time layer: Kafka streams for recent user actions. Updates candidate pool in real-time. Nearest-neighbor search on embeddings (ANN via FAISS/ScaNN).
  • Serving layer: Pre-computed recommendations cached in Redis. Real-time re-ranking based on context (time of day, device, activity).
Analysis: Spotify is known for Discover Weekly and AI DJ. Mention their public research: they use a combination of collaborative filtering (BaRT -- Bandits for Recommendations as Treatments) and NLP on podcast transcripts. Key tradeoff: exploration vs. exploitation -- do you show users safe recommendations they'll probably like, or risky ones that could expand their taste? Discussing this tradeoff shows product thinking.
Q Design a collaborative playlist system with real-time sync Hard ▾

Suggested Answer

Requirements

Multiple users can add/remove/reorder songs in a shared playlist simultaneously. Changes should propagate in near-real-time (<1s). Must handle conflict resolution gracefully.

Key Challenges

  • Concurrent edits: Two users reorder at the same time. Need conflict resolution.
  • Consistency model: Eventual consistency is acceptable -- users can tolerate brief inconsistency.
  • Scale: 5B+ playlists, but most collaborative playlists have <10 editors, so write contention is low.

Approach: CRDT-based

  • Model playlist as a Sequence CRDT (e.g., RGA -- Replicated Growable Array). Each song gets a unique position identifier that allows insert-between without conflicts.
  • Operations: InsertAfter(posId, songId), Delete(posId). Both are commutative and idempotent.
  • Changes published to Kafka topic per playlist. Subscribers (connected clients) receive updates via WebSocket.

Simpler Alternative: Last-Write-Wins with versioning

  • Each playlist has a version number. Edits include the version they're based on. If version mismatch, reject and force client to re-fetch. Simple, but causes more user-visible conflicts.
Analysis: This tests distributed systems depth. Don't over-engineer: acknowledge that most collaborative playlists have very low write contention, so even a simple optimistic locking approach works. But show you know CRDTs exist and when they'd be necessary (e.g., Spotify Blend where two users' preferences are merged). Discuss the tradeoff between CRDT complexity and operational simplicity.
Q Design an image upload system for playlist covers Medium ▾

Suggested Answer

Flow

  1. Client requests a pre-signed upload URL from the API.
  2. Client uploads image directly to object storage (GCS/S3) using the pre-signed URL -- bypasses your servers for large payloads.
  3. Object storage triggers an event (GCS notification / S3 event).
  4. Image Processing Service consumes the event: validates format, generates thumbnails (small, medium, large), applies content moderation (ML model for NSFW detection).
  5. Processed images stored in CDN-backed storage. Playlist metadata updated with image URLs.

Scale Considerations

  • Pre-signed URLs avoid your servers becoming a bottleneck for uploads.
  • Async processing via message queue decouples upload from processing.
  • Content moderation can be a separate pipeline -- don't block the upload on it.
  • Cache invalidation: when a user changes their playlist cover, CDN edges may serve stale images. Use versioned URLs (e.g., image_v3.jpg) to avoid cache invalidation entirely.
Analysis: This exact question has been reported in Spotify interviews. It tests practical system design rather than algorithmic thinking. Key insight: the pre-signed URL pattern is the standard for user-uploaded content at scale. Follow-up questions may include: "What if millions of users upload at once?" (rate limiting, queue backpressure) and "How do you handle inappropriate content?" (async moderation pipeline).
Q Design a podcast streaming and search system Hard ▾

Suggested Answer

Differences from Music Streaming

  • Podcasts are much longer (30-180 min vs 3-5 min). Seeking and resume-from-position are critical.
  • New episodes published frequently -- need an ingestion pipeline.
  • Transcription enables full-text search within episodes.
  • Dynamic ad insertion based on listener demographics and position in episode.

Architecture Additions

  • RSS Ingestion Service: Polls podcast RSS feeds periodically. Downloads new episodes, stores audio in object storage, extracts metadata.
  • Transcription Pipeline: Async ASR (Automatic Speech Recognition) via Whisper or similar. Stores timestamped transcripts in Elasticsearch for search.
  • Search: Index on title, description, and transcript. Highlight matching segments with timestamps so users can jump to relevant parts.
  • Playback State Service: Track per-user, per-episode position. Sync across devices. Redis with persistence for fast reads.
  • Ad Insertion: Server-side ad stitching (SSAI). Pre-computed ad break points. Personalized ads based on user profile.
Analysis: Podcasts are a huge part of Spotify's strategy. This question tests whether you can extend a music-streaming design to handle different content characteristics. Key insight: the transcription-based search is what differentiates a good answer. Spotify actually acquired podcast transcription technology. Discuss the cost tradeoff of transcribing all episodes vs. on-demand.

Onsite: Case Study / Production Troubleshooting

60 min Interactive dialogue Unique to Spotify

The most distinctive round in Spotify's process. You simulate being the on-call engineer responding to a production incident. The interviewer plays the role of the system -- you ask questions, request metrics, propose hypotheses, and drive the investigation. They evaluate your debugging methodology, communication under pressure, and ability to prioritize.

Tip: Practice a structured triage framework: (1) Assess impact/severity, (2) Check recent deployments and changes, (3) Review metrics and dashboards, (4) Form hypotheses and narrow down, (5) Mitigate, then root-cause. Communicate clearly about what you're doing and why at each step.
Q Scenario: Ads are not displaying for US users but EU users are unaffected Hard ▾

Suggested Approach

Step 1: Assess Impact

"How long has this been happening? What percentage of US users are affected? Is ad revenue actively being lost? Should we consider rolling back anything?"

Step 2: Narrow the Scope

  • "Is it all ad types or specific formats (audio, display, video)?"
  • "Is it all US regions or specific datacenters?"
  • "Were there any recent deployments to the ad-serving service?"

Step 3: Hypothesize

  • H1: Network/latency issue. If the ads DB is in EU, US requests may be timing out. Check: ad service latency metrics, P99 for US vs EU.
  • H2: Regional configuration drift. Feature flag or config specific to US region is misconfigured. Check: config management system, recent config changes.
  • H3: CDN/edge cache issue. US CDN nodes caching a bad response. Check: CDN cache headers, purge and test.
  • H4: Database replication lag. If US reads from a replica and the replica is lagging or down. Check: replication status, replica health.

Step 4: The Likely Answer

Based on reports, the expected answer involves database latency: the ads database is in EU, and US servers are experiencing high latency querying across the Atlantic. The fix involves either: adding a read replica in the US region, implementing caching for ad inventory, or adjusting timeout configurations.

Step 5: Mitigation & Communication

Communicate to stakeholders: "We've identified a latency issue with ad serving for US users. Impact is [X]% of ad impressions. Short-term mitigation: increasing timeout or enabling cached ad fallback. Long-term fix: deploying a US read replica."

Analysis: This exact scenario has been reported by candidates. The interviewer wants to see a structured approach, not a lucky guess. Ask questions before jumping to solutions. Key signals they look for: (1) prioritizing impact assessment, (2) systematic hypothesis elimination, (3) clear communication throughout, (4) distinguishing mitigation from root-cause fix. Even if your first hypothesis is wrong, a good process impresses more than guessing right.
Q Scenario: Playlist saves are failing for 5% of users Hard ▾

Suggested Approach

Information Gathering

  • "What error are users seeing? Client-side error or server error (4xx vs 5xx)?"
  • "Is the 5% random or correlated to something (account age, playlist size, region, client version)?"
  • "When did this start? Gradual degradation or sudden onset?"
  • "Are other playlist operations (add song, delete) also affected?"

Investigation Path

  1. Check error logs: Look for patterns in the 5xx responses. Are they timeouts, constraint violations, or something else?
  2. Check database: Is the playlist table hitting a shard hotspot? Are some shards unhealthy? Is there a specific partition that's overloaded (e.g., users with very large playlists)?
  3. Check for data issues: Could these users have corrupted playlist data? Maybe a migration ran partially.
  4. Check resource limits: Connection pool exhaustion, memory pressure, disk space on specific nodes.

Likely Root Causes

  • Playlist size limit: Users hitting an undocumented size limit (e.g., playlists with 10K+ songs hitting a row size limit in the DB).
  • Partial migration failure: A schema migration ran on most shards but failed on one, causing writes to fail for users on that shard.

Q System Design: Design a subscription billing system that handles plan changes, proration, and dunning Hard ▾

Suggested Answer

Requirements

Functional: Create/upgrade/downgrade/cancel subscriptions. Generate invoices. Charge recurring payments. Handle failed payments with retries (dunning). Prorate charges on plan changes. Support multiple currencies.

Non-functional: Exactly-once billing (never double-charge). Full audit trail. 99.99% availability. PCI DSS compliance for payment data.

Core Components

  • Subscription Service: Manages subscription lifecycle (active, paused, cancelled, past_due). Stores plan, billing cycle anchor date, current period start/end. Immutable state transitions with event sourcing.
  • Billing Engine: Scheduler scans subscriptions due for renewal (cron or event-driven). Generates invoice line items, applies proration credits/debits, calculates tax. Publishes invoice.created event.
  • Payment Orchestrator: Receives invoice, selects payment provider based on user's payment method, routes charge request. Handles provider-specific APIs behind a unified interface (Adapter pattern). Implements idempotency keys to prevent double charges on retries.
  • Ledger Service: Double-entry accounting. Every charge creates a debit (user owes) and credit (revenue recognized). Append-only -- never mutate historical entries. Enables reconciliation against external bank statements.
  • Dunning Engine: On payment failure: retry with exponential backoff (day 1, 3, 5, 7). Send dunning emails. After max retries, transition subscription to past_due, then cancelled. Allow grace period for card updates.

Proration Logic

// Upgrade mid-cycle: Premium ($9.99) -> Family ($15.99)
// 15 days into a 30-day cycle

days_remaining = 15
daily_rate_old = 9.99 / 30   = $0.333
daily_rate_new = 15.99 / 30  = $0.533

credit = days_remaining * daily_rate_old = $5.00  // unused old plan
charge = days_remaining * daily_rate_new = $8.00  // remaining new plan
net_charge = charge - credit = $3.00              // immediate charge
Analysis: This is the single most relevant system design question for this role. Spotify's Commerce Platform literally builds this system. Key signals: (1) double-entry ledger shows you understand financial systems, (2) idempotency keys show you understand distributed payment safety, (3) proration logic shows you can handle real business complexity. Discuss the tradeoff between immediate proration (complex, better UX) vs. next-cycle adjustment (simpler, delayed).
Q System Design: Design a payment orchestration layer that supports 16+ payment providers Hard ▾

Suggested Answer

The Problem

Spotify operates in 180+ markets with providers like Stripe, PayPal, Klarna, Boku (carrier billing), Apple IAP, Google Play Billing, and local providers (iDEAL in NL, Swish in SE, PIX in BR). Each has different APIs, webhook formats, settlement timelines, and failure modes.

Architecture

  • Provider Adapter Layer: Each provider implements a common interface: authorize(), capture(), refund(), webhook(). Adapters translate between your domain model and provider-specific APIs. New providers = new adapter, no core changes.
  • Routing Engine: Selects the optimal provider based on: payment method type, user's country, provider availability (circuit breaker status), historical success rate, transaction fees. Supports fallback: if primary provider fails, automatically retry with secondary.
  • Idempotency Layer: Every payment attempt has a unique idempotency key stored in the DB. Before processing, check if this key was already processed. If yes, return the cached result. Prevents double-charging on network retries.
  • Webhook Normalizer: Each provider sends webhooks in different formats and with different delivery guarantees. Normalize into internal events: payment.succeeded, payment.failed, refund.completed. Deduplicate based on provider transaction ID.
  • Reconciliation Service: Daily batch job compares internal ledger against provider settlement reports. Flags discrepancies for manual review. Handles edge cases: chargebacks, partial settlements, currency conversion differences.

Key Design Decisions

  • Two-phase capture: Authorize first (reserve funds), capture later (after service provisioning). If provisioning fails, release the authorization instead of refunding.
  • Provider health monitoring: Circuit breaker per provider. If Stripe's success rate drops below 95%, route new payments to Adyen. Alert on-call.
  • PCI scope minimization: Never store raw card numbers. Use provider tokenization. Your systems only handle tokens, reducing PCI DSS audit scope.
Analysis: Spotify literally built this -- their Billing API abstracts 16+ providers behind a single interface. Mentioning Spotify's Checkout API and Billing API by name shows you've researched the team. Key insight: the routing engine with fallback is what makes multi-provider worth the complexity. Q How do you ensure exactly-once payment processing in a distributed system? Hard ▾

Suggested Answer

True exactly-once is impossible in distributed systems. What we achieve is effectively-once through idempotency.

Implementation

// Client generates a unique idempotency key per payment intent
POST /v1/charges
Idempotency-Key: idem_abc123xyz

// Server-side:
1. BEGIN TRANSACTION
2. Check idempotency_keys table for idem_abc123xyz
3. If found -> return cached response (no re-processing)
4. If not found -> process payment
5. Store result in idempotency_keys table
6. COMMIT TRANSACTION

Critical details:

  • The idempotency check and payment processing must be in the same database transaction. Otherwise, a crash between check and insert creates a window for double-processing.
  • Idempotency keys should expire (e.g., 24-48 hours). Stale keys waste storage and complicate debugging.
  • For async flows (e.g., Kafka consumers), use the Outbox pattern: write the payment result and the outgoing event to the same DB transaction, then a separate process publishes from the outbox table.
  • Downstream side effects (sending receipt email, updating analytics) must also be idempotent or gated behind the payment's unique ID.

What About Saga?

For multi-step payment flows (authorize -> provision -> capture), use the Saga pattern with compensating transactions. If provisioning fails after authorization, the compensating action is to void the authorization.

Analysis: This is the most important concept for a Commerce Platform engineer. Double-charging a user is a P0 incident. Key signal: explaining that the idempotency check and write must be atomic (same transaction). Many candidates miss this and describe a check-then-act pattern with a race condition window. Also mention: Stripe's idempotency implementation is public -- referencing it shows domain knowledge.
Q Explain double-entry accounting and why it matters for a payment backend Medium ▾

Suggested Answer

Double-entry accounting records every financial event as two entries: a debit and a credit of equal amounts. The sum of all debits must always equal the sum of all credits (the accounting equation).

Example: User pays $9.99 for Premium

Transaction: txn_001 (2026-05-14)
+-------------------------------+--------+--------+
| Account                       | Debit  | Credit |
+-------------------------------+--------+--------+
| Accounts Receivable (user)    |        | $9.99  |
| Revenue (subscription)        | $9.99  |        |
+-------------------------------+--------+--------+
// Debit total = Credit total = $9.99  ✓

Example: Refund $3.00 proration credit

Transaction: txn_002
+-------------------------------+--------+--------+
| Account                       | Debit  | Credit |
+-------------------------------+--------+--------+
| Revenue (subscription)        |        | $3.00  |
| Accounts Receivable (user)    | $3.00  |        |
+-------------------------------+--------+--------+

Why it matters:

  • Self-auditing: If debits != credits, you know something is wrong. Built-in error detection.
  • Immutable audit trail: Append-only ledger. Corrections are new entries, not mutations. Required for financial compliance and audit.
  • Reconciliation: You can reconcile your internal ledger against external provider statements by comparing transaction-level entries.
  • Revenue recognition: Critical for SaaS -- when does Spotify "recognize" revenue for a yearly subscription? Monthly as service is delivered, not upfront.
Analysis: Many backend engineers have never worked with financial systems. Demonstrating knowledge of double-entry accounting immediately sets you apart for a Commerce Platform role. Key insight: the ledger is append-only. In traditional CRUD, you'd update a balance field. In financial systems, you insert journal entries and compute balances from the sum of entries. This is fundamental for audit compliance.
Q How would you modernize a legacy billing system while keeping it running in production? Hard ▾

Suggested Answer

Approach

Q How do you design a reconciliation system between your ledger and external payment providers? Medium ▾

Suggested Answer

Reconciliation is the process of verifying that your internal records match external provider records. It's the last line of defense against financial discrepancies.

Architecture

  • Ingestion: Download settlement reports from each provider (daily/weekly). Each provider has a different format -- CSV, API, SFTP. Normalize into a common schema: (transaction_id, amount, currency, status, timestamp).
  • Matching: Join internal ledger entries with provider records on transaction ID. Classify each record as: matched (amounts agree), mismatched (amounts differ), internal-only (we recorded it, provider didn't), external-only (provider recorded it, we didn't).
  • Resolution: Matched records need no action. Mismatched records go to a review queue with context (which side is likely correct?). Internal-only might indicate a provider webhook we missed. External-only might indicate a charge that bypassed our system (dangerous -- investigate immediately).
  • Alerting: If unmatched records exceed a threshold (e.g., >0.1% of daily transactions), page the on-call engineer. Track reconciliation health as a key metric.

Timing Challenges

Settlements aren't instant. A charge on Monday might settle on Wednesday. Your reconciliation window must account for settlement delays. Use a rolling window: reconcile transactions from 3-5 days ago, not today.

Analysis: Reconciliation is unglamorous but critical. Many startups skip it and discover discrepancies during audits. For a Commerce Platform role, showing you understand the full payment lifecycle (charge -> settle -> reconcile) demonstrates maturity. Key insight: external-only records are the scariest category -- they mean money moved without your system knowing about it.

Q How do you handle multi-currency pricing and avoid floating-point errors in financial calculations? Medium ▾

Suggested Answer

Rule #1: Never use float or double for money. Use integer cents (or the smallest currency unit) or a dedicated decimal type.

// BAD - floating point
double price = 9.99;
double tax = price * 0.20;  // 1.998000000000002

// GOOD - integer cents (Java)
long priceInCents = 999;
long taxInCents = priceInCents * 20 / 100;  // 199

// GOOD - BigDecimal (Java)
BigDecimal price = new BigDecimal("9.99");
BigDecimal tax = price.multiply(new BigDecimal("0.20"))
    .setScale(2, RoundingMode.HALF_UP);  // 2.00

Multi-Currency Architecture

  • Store in user's currency: Prices are set per market (e.g., $9.99 USD, 109 SEK, R$34.99 BRL). Don't convert at charge time -- use pre-set prices.
  • Always store currency code alongside amount: (amount: 999, currency: "USD"). Never have a bare amount column.
  • Conversion for reporting: When aggregating revenue across currencies for financial reporting, convert using the exchange rate at the time of the transaction, not today's rate. Store the rate used.
  • Zero-decimal currencies: JPY, KRW, VND don't have subunits. 500 JPY = 500, not 50000. Your system must handle different currency exponents.
Analysis: Spotify operates in 180+ markets with 60+ currencies. Floating-point bugs in financial calculations can cause real monetary discrepancies at scale. Step 2: Investigate

  • Check the idempotency layer: Are idempotency keys being generated correctly? A bug in key generation (e.g., using a timestamp that changes on retry) would cause retries to be treated as new charges.
  • Check the billing scheduler: Did the cron job fire twice? Was there a deploy that restarted the scheduler mid-cycle, causing it to re-process users it already billed?
  • Check provider webhooks: Are we receiving duplicate payment.succeeded webhooks and processing both? Webhook deduplication might be broken.
  • Check for race conditions: Are two instances of the billing service processing the same user concurrently due to inadequate distributed locking?

Step 3: Mitigate Immediately

  • Pause the billing scheduler to stop new double charges.
  • Identify all affected users by querying for users with two charges in the current billing period.
  • Issue automated refunds for the duplicate charges.
  • Send proactive communication to affected users before they notice.
Analysis: Double-charging is the nightmare scenario for any payment team. The response here tests both technical debugging and incident management maturity. Key signals: (1) immediately quantifying impact, (2) pausing the scheduler to stop the bleeding before root-causing, (3) proactive user communication. Mentioning refund automation shows you think about the full resolution, not just the technical fix.
