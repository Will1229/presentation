Deep dive: HTTP Range Requests
HTTP range requests let the client fetch a specific byte range of a file using the Range: bytes=0-65535 header. The server responds with 206 Partial Content and only the requested chunk. This is how audio streaming works without a custom protocol:
  • The client requests the first chunk to start playback immediately (low time-to-first-byte)
  • Subsequent chunks are fetched as the buffer drains, allowing seek by requesting arbitrary byte offsets
  • CDNs support range requests natively -- no application server needed in the streaming path
  • The client can pause, resume, or seek without re-downloading the entire file
At MVP scale this is simpler than HLS/DASH segment-based streaming, which requires pre-segmenting files and a manifest. Range requests work with plain MP3 files stored on S3.
Deep dive: PostgreSQL Full-Text Search
Postgres FTS uses tsvector (document) and tsquery (query) types with GIN indexes for fast lookups. For the MVP:
  • Create a tsvector column on the songs table combining title, artist name, and album title
  • Add a GIN index: CREATE INDEX idx_songs_fts ON songs USING GIN(search_vector)
  • Query with: WHERE search_vector @@ plainto_tsquery('english', 'user input')
  • Supports stemming (searching "running" matches "run"), stop-word removal, and ranking via ts_rank()
Limitations: No fuzzy matching (typos), no phonetic matching, no per-field boosting, and performance degrades past ~500K rows with complex queries. These gaps push us to Elasticsearch in Stage 2.
Tradeoff: Single audio quality
Single bitrate (128kbps) means simpler storage and no adaptive bitrate logic. Users on slow connections will experience buffering -- acceptable at MVP scale where the goal is validating product-market fit, not perfecting the streaming experience.

Intentionally skipped: recommendations, offline mode, social features, multiple audio qualities, real-time lyrics, podcasts.

Interview Questions
Why not start with microservices from day one?
architecture
At 0-10K users, the overhead of microservices (service discovery, distributed tracing, network latency, deployment orchestration) far outweighs the benefits. A monolith lets a small team iterate fast with simple deploys. Premature decomposition is harder to reverse than a delayed split.
How would you handle user authentication?
security
JWT with short-lived access tokens (15 min) and longer-lived refresh tokens (7 days) stored in httpOnly cookies. Stateless auth means no session store. For MVP, email + password with bcrypt is sufficient. OAuth/social login comes later as a growth lever.
Why PostgreSQL over NoSQL for the catalog?
data modeling
The data is inherently relational -- songs belong to albums, albums to artists, playlists reference songs with ordering. PostgreSQL gives joins, foreign keys, transactions, and full-text search out of the box. NoSQL would require denormalization and manual consistency for no benefit at this scale.

Tradeoff: Elasticsearch vs Algolia
Elasticsearch: Self-managed, full control over relevance tuning, no per-query cost. But requires cluster management, index tuning, and handling sync lag.

Algolia: Managed, simpler, great DX. But expensive at scale and creates vendor lock-in.

At this stage, Elasticsearch is the better bet -- the team has infra skills, and the cost savings matter more than DX convenience.
Deep dive: Song Caching -- CDN vs Redis vs Local Disk vs Varnish
A music streaming system has two distinct caching problems: metadata (song info, playlists, user profiles) and audio files (the actual MP3/AAC bytes). Each needs a different solution.

The right combination for a streaming service:
  • Audio files: CDN for delivery + tiered origin storage (hot on SSD, cold on cheap object storage)
  • Metadata: Redis for hot paths (popular songs, active sessions, playlist reads)
  • API responses: Varnish/Nginx for public endpoints that don't vary per user (artist pages, album pages, charts)
  • Client-side: Don't forget browser/app cache -- cache song metadata aggressively, prefetch the next song in the playlist
Deep dive: Change Data Capture (CDC)
CDC captures row-level changes from the database transaction log (Postgres WAL) and streams them to downstream consumers. This is how we keep Elasticsearch in sync without dual-writing:
  • How it works: A CDC connector (e.g., Debezium) reads the Postgres WAL, converts each INSERT/UPDATE/DELETE into a change event, and publishes it to Kafka
  • Consumer side: An ES indexer consumes these events and applies them to the search index. Typical lag: 1-5 seconds
  • Why not dual-write? If the app writes to both Postgres and ES, a crash between the two writes leaves them inconsistent. CDC is transactional -- if it's in the WAL, it will be captured
  • Schema evolution: Debezium supports schema registry integration, so index mapping changes can be coordinated with DB migrations
  • Operational cost: Adds Kafka + Debezium + a consumer service. Worth it because search consistency directly affects user trust
Deep dive: Payment System with Stripe
At PMF stage, the goal is validating revenue -- not building payment infrastructure. Stripe handles the hard parts:
  • Stripe Billing: Manages subscription lifecycle -- trial periods, upgrades/downgrades, proration, invoicing, and dunning (failed payment retries)
  • Stripe Checkout: Hosted payment page handles PCI compliance. You never touch raw card numbers -- Stripe tokenizes everything client-side
  • Webhooks: Stripe pushes events (invoice.paid, customer.subscription.deleted, payment_intent.payment_failed) to your endpoint. These update the local subscriptions table
  • Entitlement flow: Streaming service checks subscriptions.status and current_period_end before granting premium quality. Cache this in Redis with a short TTL (60s) to avoid DB lookups on every play

Interview Questions
How do you keep Elasticsearch in sync with PostgreSQL?
data consistency
CDC (Change Data Capture) via Debezium reads the Postgres WAL and streams changes to ES -- reliable but adds infra. Dual-write from the application is simpler but risks inconsistency if one write fails. CDC is worth the investment because search consistency directly affects user trust. Accept seconds of lag.
When would you use a cache vs a read replica?
scaling
Read replicas give consistent reads (just delayed) and handle complex queries. Redis cache gives sub-millisecond reads but requires invalidation logic. Use replicas for browse/search queries that tolerate slight lag. Use Redis for hot data read thousands of times per second (popular song metadata, sessions). They complement each other.
How does adaptive bitrate streaming work?
streaming
Encode each song at multiple bitrates (64/128/256 kbps). Client measures available bandwidth and buffer state, then requests the appropriate quality. Can use HTTP range requests with quality selection at playback start, or segment-based (HLS/DASH) for mid-stream adaptation. Quality selection at start is simpler and sufficient here.
Why not cache audio files in Redis?
caching
Cost. A song is 3-8 MB; 10M songs = 30-80 TB. Redis costs ~$3,000/TB/month vs $23/TB/month in S3. Redis is for metadata (song info, sessions, playlists) where sub-millisecond latency matters on small objects. Audio delivery belongs on CDN edge nodes, which are optimized for large file serving with range request support. Different cache layers for different data shapes.
How do you handle subscription entitlements?
payments
Stripe is the payment processor, but your DB is the source of truth for entitlements. Stripe webhooks update the local subscriptions table. The streaming service checks status and current_period_end before granting premium quality. Cache this in Redis (60s TTL) to avoid a DB hit on every play. Webhook failures are retried by Stripe with exponential backoff -- design the handler to be idempotent.

Deep dive: Payment Service Isolation
At the microservices stage, payment logic moves out of the monolith into a dedicated Payment Service. This isn't just an architectural preference -- it's a compliance requirement:
  • PCI scope reduction: Only the Payment Service touches Stripe tokens and subscription state. Other services query entitlements via gRPC (IsUserPremium(user_id)) -- they never see payment data
  • Isolated database: Payment DB is on a separate Postgres cluster with stricter access controls, encryption at rest, and audit logging. No other service has network access to this DB
  • Entitlement events: When a subscription changes, the Payment Service publishes events to Kafka (subscription.activated, subscription.cancelled, subscription.past_due). The Streaming Service, Ad Service, and Download Service consume these to update their behavior
  • Plan complexity: Family plans (up to 6 members), student plans (verification flow), annual vs monthly, regional pricing, promotional trials -- this logic is complex enough to deserve its own service and team
  • Webhook reliability: Stripe webhook handler uses an idempotent inbox pattern -- store the event ID before processing, skip duplicates. Failed processing goes to a dead letter queue with alerting
Deep dive: gRPC for Inter-Service Communication
When services talk to each other, REST/JSON works but has overhead: text serialization, no built-in schema enforcement, and HTTP/1.1 connection limits. gRPC solves these at the microservices stage:
  • Protobuf serialization: Binary format, 3-10x smaller than JSON, faster to serialize/deserialize. Matters when services fan out 5+ calls per user request
  • HTTP/2 multiplexing: Multiple RPC calls share a single TCP connection, eliminating head-of-line blocking and reducing connection overhead
  • Schema-first contracts: .proto files are the source of truth. Breaking changes are caught at compile time, not in production
  • Streaming: gRPC supports server-streaming (useful for the recommendation service pushing updates) and bidirectional streaming
  • Where NOT to use it: Keep REST/JSON for the public-facing API (browser compatibility, developer DX, cURL-friendly). gRPC is internal only

Tradeoff: Playlist DB -- Postgres vs Cassandra
Collaborative playlists with many concurrent editors create hot rows in Postgres. Cassandra handles write-heavy workloads but makes ordered data (playlist position) harder.

Decision heuristic: If collaborative playlists become a top-3 feature by usage, migrate to Cassandra with position encoded as a sortable string (e.g., fractional indexing). Otherwise, Postgres with row-level locking is fine.
Tradeoff: Sync vs async for cross-service data
  • API calls: Simpler, strong consistency, but runtime coupling -- one service down cascades to callers.
  • Events: Decoupled, resilient, but eventual consistency -- users might see stale data briefly.
Rule of thumb: Reads that tolerate staleness (catalog browsing, recommendations) use events. Reads that need freshness (account state, payment status) use synchronous APIs.
Tradeoff: Recommendation complexity
Collaborative filtering is the starting point but has a cold-start problem for new users (no listening history to base recommendations on).

Solution: Hybrid approach -- content-based features (genre, tempo, mood) fill the cold-start gap. Full ML pipeline (training infra, model serving, A/B testing) is expensive to build and maintain. Start with batch-computed recommendations, add real-time personalization only when A/B tests show it moves retention.
Interview Questions
How do you decide where to split services?
architecture
Follow domain boundaries, not technical layers. Each service maps to a business capability owned by one team. Key heuristics: can it deploy independently? Does it own its data? Does changing it require coordinating with other teams? If services share a database or must deploy together, the boundary is wrong.
What are the failure modes of Kafka?
reliability
Message loss (mitigated by acks=all, replication factor 3), consumer lag (monitoring + auto-scaling), ordering violations (partition key by entity ID), duplicate delivery (idempotent consumers with dedup keys), poison messages (dead letter queues). Design every consumer to be idempotent -- the single most important pattern.
How do you handle the recommendation cold start?
MLpersonalization
Three layers. New users: popularity-based (trending, top charts). After onboarding survey (3-5 genre picks): content-based filtering using audio features. After ~50 plays: collaborative filtering using similar users' listening patterns. Blend all three with weights that shift as user history grows.
Why gRPC internally but REST externally?
protocols
Internal services benefit from Protobuf's compact binary format and HTTP/2 multiplexing -- 3-10x less bandwidth and lower latency on service fan-outs. But the public API stays REST/JSON: browsers can't call gRPC natively, third-party developers expect REST, and debugging with cURL matters. The API gateway translates between the two.
Why isolate the Payment Service from other microservices?
paymentscompliance
PCI DSS scope. Any service that can access payment data must meet PCI requirements -- audits, penetration testing, access controls. By isolating the Payment Service with its own network segment and database, only that one service is in PCI scope. Other services query entitlements via a simple gRPC call (IsUserPremium) and never touch payment data. This reduces audit surface from dozens of services to one.

Deep dive: Migrating Off Stripe at Scale
At 100M+ subscribers paying $9.99/month, Stripe's 2.9% + 30c means ~$35M/year in processing fees. That justifies a dedicated payments engineering team:
  • Phase 1 -- Direct acquirer relationships: Negotiate directly with payment processors (Adyen, Worldpay) for lower per-transaction rates (0.5-1.5% vs 2.9%). Stripe stays as a fallback for unsupported regions
  • Phase 2 -- Regional payment gateways: Different regions have dominant payment methods. EU needs SEPA Direct Debit and iDEAL. Southeast Asia needs GrabPay and GCash. Latin America needs boleto and PIX. Each requires a regional gateway integration
  • Phase 3 -- Card vault: Build an in-house PCI Level 1 card vault to store tokenized card data. This removes the dependency on Stripe for recurring charges but requires a dedicated security team, annual PCI audits, and SOC 2 compliance
  • Keep Stripe for: New market launches (Stripe is faster to deploy), dispute management (their tooling is excellent), and as a fallback processor for redundancy
The migration is gradual, not a cutover. Run Stripe and the new processor in parallel. Route 5% of new subscriptions to the new processor, monitor chargeback rates and payment success rates, then increase the split. Never migrate existing subscribers' stored cards without explicit consent.
Tradeoff: Strong vs eventual consistency for user data
Spanner/CockroachDB: Global strong consistency but higher latency per write (cross-region quorum).

Kafka replication: Faster writes, but users might see stale data briefly when traveling.

Pragmatic split: Most user actions (play, browse, add to playlist) tolerate staleness. Payments, account changes, and subscription state need strong consistency. Use Spanner for the critical path, Kafka for everything else.
Tradeoff: Cell isolation vs data locality
Fully isolated cells are safer but mean a user traveling abroad hits a "foreign" cell with no local data.

Solutions:
  • Read replicas of user data in all regions (high storage cost, low read latency)
  • Sticky routing by home region with fallback to nearest cell (lower cost, higher latency for travelers)
  • Hybrid: replicate hot user data (recent playlists, preferences), lazy-load cold data on demand
Tradeoff: Monorepo vs polyrepo at 100+ services
Monorepo: Atomic cross-service changes, easier refactoring, single CI config. But CI gets slow, and every engineer clones everything.

Polyrepo: Team autonomy, fast CI per service, independent release cycles. But cross-service changes require coordinated PRs.

Most companies at this scale land on monorepo with service-level CI boundaries (e.g., Bazel, Nx). The coordination overhead of polyrepo outweighs the CI challenges of monorepo.
Interview Questions
How do you decide what needs strong consistency?
distributed systems
Map each operation to its business impact if stale. Playing a song with a 2-second-old playlist? Fine. Processing a payment with stale subscription state? Dangerous. Rule: anything involving money, account security, or legal compliance gets strong consistency (Spanner). Everything else uses eventual consistency.
How would you handle a full region outage?
reliability
Cell-based architecture means a region failure is contained. GeoDNS health checks detect the outage and reroute traffic to the nearest healthy region. User data is either replicated (hot) or fetched on-demand (cold). Targets: RTO < 5 minutes (DNS TTL), RPO < 30 seconds (Kafka replication lag).
What's your sharding strategy for user data?
scaling
Hash user_id with consistent hashing to assign users to shards. Avoids hotspots from sequential IDs. Each shard is a Postgres cluster with primary + replicas. Catalog data is not sharded -- replicated globally since it's read-heavy, rarely written. For cross-shard queries (friend activity), use fan-out or denormalized views.
When does building in-house payment processing make sense?
paymentscost
When Stripe fees exceed the fully loaded cost of a payments engineering team (5-10 engineers + PCI audits + compliance). At ~$10M/year in Stripe fees, the economics start to favor in-house. But it's not all-or-nothing: start by negotiating lower rates with direct acquirers (Adyen, Worldpay) while keeping Stripe as fallback. Build the card vault only when you have the security team to maintain PCI Level 1 certification year-round.
